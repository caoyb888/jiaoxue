## 1. 存储架构总览

### 1.1 分库分域策略

系统按业务域拆分数据库，不同类型数据使用最适配的存储介质：

| 存储介质 | 库/集合名 | 承载域 | 说明 |
|----------|-----------|--------|------|
| MySQL 8.0（主从） | `edu_db` | 用户、课程、互动、题库、测试、成绩 | 事务性业务主数据，主写从读 |
| MySQL 8.0（主从） | `edu_audit_db` | 操作日志 | 独立审计库，180天留存，等保三级 |
| MySQL 8.0 | `edu_jwxt_db` | 教务对接 | 教务同步暂存，ETL中间层 |
| ClickHouse | `edu_analytics` | 统计分析、大屏 | OLAP，Kafka 批量写入 |
| MongoDB | `edu_ai` | AI对话、课堂转写、AI批改、思维导图、分组讨论 | 非结构化，动态Schema，完整覆盖5类AI数据 |
| Redis 7 Cluster | — | Session、签到队列、布隆过滤器、交卷暂存 | TTL 强制管控，见 CLAUDE.md §7.3 |
| MinIO | `edu-files` Bucket | 文件对象存储 | 含生命周期策略；元数据存 MySQL |
| Elasticsearch 8 | `edu_search` | 题目全文检索、学生搜索 | 完整 Mapping 见 §14 |

### 1.2 主业务库表域分布（V2.0 更新）

`edu_db` 中共 **58 张**表（含本次新增），按域前缀组织：

```
edu_db
├── sys_*       用户域（4张）：sys_user / sys_dept / user_role / user_wechat
├── course_*    课程域（7张）：course / class_room / class_student / class_group

```

## 2. 字段命名与注释规范

### 2.1 通用规则

```
规则 1   表名：全小写 snake_case，业务意义清晰，禁止单字母表名
规则 2   字段名：全小写 snake_case，禁止使用数据库保留字
规则 3   主键：统一命名为 id，类型 BIGINT AUTO_INCREMENT
规则 4   外键字段：{关联表单数形式}_id，如 class_id / teacher_id
规则 5   时间字段：created_at / updated_at（DATETIME，MyBatis-Plus 自动填充）
规则 6   软删除：is_deleted TINYINT DEFAULT 0（0=正常 1=已删除）
规则 7   状态字段：status TINYINT，COMMENT 必须列举所有枚举值
规则 8   布尔语义字段：is_{xxx} TINYINT DEFAULT 0（0=否 1=是）
规则 9   金额/分值：DECIMAL(n,2)，禁止用 FLOAT/DOUBLE
规则 10  手机号：VARCHAR(255) 存 AES-256 密文，禁止明文入库（详见 §3.1）
规则 11  文件大小：INT UNSIGNED（单位 KB，最大可表示约 4TB，BIGINT 浪费）
规则 12  教务对接字段（jwxt_*）：必须建独立普通索引
```

### 2.2 COMMENT 规范

**每列必须有 COMMENT，status 类字段必须枚举所有可能值：**

```sql
-- ✅ 正确
status    TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0-未开始 1-进行中 2-已结束',
user_type TINYINT NOT NULL           COMMENT '用户类型：1-学生 2-教师 3-院系管理员 4-校级管理员',

-- ❌ 错误（无 COMMENT 或描述不完整）
status    TINYINT NOT NULL DEFAULT 0,
user_type TINYINT NOT NULL COMMENT '用户类型',
```

### 2.3 存储引擎与字符集

```sql
ENGINE=InnoDB
DEFAULT CHARSET=utf8mb4
COLLATE=utf8mb4_unicode_ci
ROW_FORMAT=DYNAMIC     -- 支持大 VARCHAR/TEXT 字段前缀索引
```

## 3. 用户域 — sys_* 表

### 3.1 sys_user — 用户主表

> **V2.0 修订：手机号静态加密**  
> 审查意见指出 `phone` 字段在等保三级要求下需数据库静态加密。  
> 修订方案：**应用层 AES-256 加密**后存入，字段重命名为 `phone_cipher`，类型调整为 `VARCHAR(255)`（密文比明文长）。检索时通过加密后的密文查询，或通过 Redis 维护 `AES(phone) → user_id` 的映射缓存。

```sql
CREATE TABLE sys_user (
    id                BIGINT       NOT NULL AUTO_INCREMENT  COMMENT '用户ID，系统主键',
    student_no        VARCHAR(20)  NULL                     COMMENT '学号（学生专有，由教务系统同步，唯一）',
    username          VARCHAR(50)  NOT NULL                 COMMENT '用户名（系统登录名，唯一）',
    real_name         VARCHAR(50)  NOT NULL                 COMMENT '真实姓名',
    -- ▼ V2.0：手机号改为密文存储，字段重命名，等保三级静态加密要求
    phone_cipher      VARCHAR(255) NULL                     COMMENT '手机号 AES-256 密文（禁止存明文；Redis维护 cipher→uid 检索缓存）',
    email             VARCHAR(100) NULL                     COMMENT '邮箱地址',
    password_hash     VARCHAR(255) NULL                     COMMENT 'BCrypt 密码散列（10轮，微信/手机号登录可为NULL）',
    user_type         TINYINT      NOT NULL DEFAULT 1       COMMENT '用户类型：1-学生 2-教师 3-院系管理员 4-校级管理员',
    dept_id           BIGINT       NULL                     COMMENT '所属院系ID → sys_dept.id',
    open_id           VARCHAR(64)  NULL                     COMMENT '微信 openId（微信登录绑定，唯一）',
    union_id          VARCHAR(64)  NULL                     COMMENT '微信 unionId（跨应用唯一标识）',
    avatar_url        VARCHAR(500) NULL                     COMMENT '头像文件路径（存 MinIO）',
    status            TINYINT      NOT NULL DEFAULT 1       COMMENT '账号状态：0-禁用 1-正常 2-待审核',
    is_deleted        TINYINT      NOT NULL DEFAULT 0       COMMENT '逻辑删除：0-正常 1-已删除',
    last_login_at     DATETIME     NULL                     COMMENT '最后登录时间',
    -- ▼ 教务对接字段，V2.0 补充索引（防止全表线性扫描）
    jwxt_uid          VARCHAR(50)  NULL                     COMMENT '教务系统原始用户ID（ETL同步对照键，建独立索引）',
    archive_photo_path VARCHAR(500) NULL                    COMMENT '档案照 MinIO 加密路径（AES-256加密对象，人脸核验用）',
    created_at        DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP                     COMMENT '创建时间',
    updated_at        DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_username   (username),
    UNIQUE KEY uk_student_no (student_no),
    UNIQUE KEY uk_open_id    (open_id),
    INDEX idx_dept_id   (dept_id),
    INDEX idx_user_type (user_type),
    INDEX idx_status    (status),
    -- ▼ V2.0 新增：ETL 对照字段索引，避免全表扫描
    INDEX idx_jwxt_uid  (jwxt_uid)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='用户主表：学生/教师/管理员（手机号AES-256加密存储）';
```

**手机号加密操作规范（等保三级）：**

```
写入：AES_ENCRYPT(phone, ${AES_KEY}) → phone_cipher
检索：SELECT id FROM sys_user WHERE phone_cipher = AES_ENCRYPT(?, ${AES_KEY})
缓存：Redis Hash  key=phone:cipher:{encryptedPhone}  value=userId  TTL=24h
日志：禁止在任何日志中打印 phone_cipher 原文
```

### 4.3 class_student — 班级学生关联表

> **V2.0 修订：移除 `group_id` 字段**  
> 原设计将 `group_id` 直接存在 `class_student` 大表，每次临时分组变更都会引发该表的集中行锁更新。  
> 修订方案：`group_id` 从本表移除，分组关系由专用的 `class_group_student` 表（§4.4）管理。

```sql
CREATE TABLE class_student (
    id          BIGINT      NOT NULL AUTO_INCREMENT COMMENT '记录ID',
    class_id    BIGINT      NOT NULL                COMMENT '教学班ID → class_room.id',
    student_id  BIGINT      NOT NULL                COMMENT '学生ID → sys_user.id',
    student_no  VARCHAR(20) NOT NULL                COMMENT '学号（冗余，减少关联查询）',
    -- ▼ V2.0：移除 group_id（分组关系由 class_group_student 独立维护）
    status      TINYINT     NOT NULL DEFAULT 1      COMMENT '选课状态：0-退课 1-正常选课',
    joined_at   DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '加入时间',
    created_at  DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_class_student (class_id, student_id),
    INDEX idx_student_id (student_id),
    INDEX idx_class_id   (class_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='班级学生关联表（多对多；分组关系见 class_group_student）';
```

## 5. 互动域 — attendance / interact 表

### 5.1 attendance — 签到记录表

> **V2.0 修订：UNIQUE KEY 降级为普通复合索引**  
>
> 原问题：高并发批量落库时，`UNIQUE KEY uk_lesson_student(lesson_id, student_id)` 会引发频繁的索引锁竞争，降低批量 INSERT 吞吐。  
>
> 修订方案：
> - 移除 UNIQUE KEY，改为普通 INDEX（批量 INSERT 不触发唯一性检查锁）。
> - **幂等去重职责转移**：由 Redis 布隆过滤器（`attend:bloom:{lessonId}`）在接入层保证唯一性（详见技术方案 §4.1.3），数据库不再作为幂等层。
> - 若需事后校验数据完整性，可由 XXL-Job 定时任务扫描重复记录进行修复。

```sql
CREATE TABLE attendance (
    id           BIGINT      NOT NULL AUTO_INCREMENT COMMENT '签到记录ID',
    lesson_id    BIGINT      NOT NULL                COMMENT '课堂ID → lesson.id',
    student_id   BIGINT      NOT NULL                COMMENT '学生ID → sys_user.id',
    class_id     BIGINT      NOT NULL                COMMENT '教学班ID → class_room.id（冗余，加速统计）',
    status       TINYINT     NOT NULL DEFAULT 1      COMMENT '考勤状态：0-缺勤 1-正常签到 2-迟到 3-请假（教师修改）',
    method       VARCHAR(20) NOT NULL DEFAULT 'QR'  COMMENT '签到方式：QR-扫码 CODE-口令 MANUAL-教师手动补签',
    attended_at  DATETIME    NULL                    COMMENT '签到时间（教师手动补签可为NULL）',
    ip_address   VARCHAR(45) NULL                    COMMENT '签到IP地址（IPv4/IPv6）',
    is_modified  TINYINT     NOT NULL DEFAULT 0      COMMENT '教师是否修改过考勤：0-原始 1-已修改',
    modifier_id  BIGINT      NULL                    COMMENT '修改人ID（教师）→ sys_user.id',
    created_at   DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间（批量异步落库时间）',
    PRIMARY KEY (id),
    -- ▼ V2.0：UNIQUE KEY 降级为 INDEX，消除批量 INSERT 时的唯一索引锁竞争
    -- 幂等职责由 Redis BloomFilter 在应用层承担（详见 CLAUDE.md §8.1）
    INDEX idx_lesson_student (lesson_id, student_id),
    INDEX idx_lesson_id      (lesson_id),
    INDEX idx_student_id     (student_id),
    INDEX idx_class_id       (class_id),
    INDEX idx_attended_at    (attended_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='签到记录表（异步批量落库；唯一性由 Redis BloomFilter 保证，非 DB UNIQUE 约束）';
```

**签到高并发落库架构（补充说明）：**

```
学生扫码
  │
  ├─① Redis BloomFilter 去重（O(1)，幂等保障层）
  │    key: attend:bloom:{lessonId}  TTL: 课堂结束后24h
  │
  ├─② 写 Redis List 队列（O(1)，毫秒响应）
  │    key: attend:queue:{lessonId}
  │
  ├─③ Redis Counter +1 → WebSocket 广播教师端
  │
  └─④ Consumer 每 500ms 批量 INSERT（无唯一索引，高吞吐）
       INSERT INTO attendance (...) VALUES (...),(...),...
       → 一次 SQL 插入 ≤ 50 条
```

### 7.6 exam_submit_queue — 交卷流水暂存表（新增）

> **V2.0 新增**：专门承载高并发交卷容灾与打散策略的数据库层支撑。  
>
> **设计背景**：当 3000 名学生在 30 秒内集中交卷时，若直接写入 `student_answer` 表将导致数据库瘫痪（写入量 ≈ 3000 人 × 平均 20 题 = 60000 行/30s）。  
>
> **分层策略**：
> - **Layer 1（前端）**：IndexedDB 实时草稿 + 学号取模打散 0~30s 延迟提交
> - **Layer 2（接口层）**：先写 Redis 暂存（毫秒响应），发 Kafka 消息  
> - **Layer 3（本表）**：接收 Kafka 消费者的平滑写入，按行粒度暂存完整交卷快照
> - **Layer 4（异步处理）**：逐条解析展开到 `student_answer` 明细表

```sql
CREATE TABLE exam_submit_queue (
    id              BIGINT       NOT NULL AUTO_INCREMENT COMMENT '流水ID',
    publish_id      BIGINT       NOT NULL                COMMENT '发布ID → exam_publish.id',
    student_id      BIGINT       NOT NULL                COMMENT '学生ID → sys_user.id',
    -- 完整交卷快照（JSON格式，一次性存储所有题目的作答）
    answers_json    MEDIUMTEXT   NOT NULL                COMMENT '答题快照 JSON（格式见下方说明）',
    submit_type     VARCHAR(20)  NOT NULL DEFAULT 'MANUAL' COMMENT '交卷类型：MANUAL-主动交卷 AUTO-倒计时自动交卷 FORCE-教师强制收卷',
    client_submit_at DATETIME    NOT NULL                COMMENT '客户端提交时间（学号打散后的实际触发时间）',
    process_status  TINYINT      NOT NULL DEFAULT 0      COMMENT '处理状态：0-待处理 1-处理中 2-已展开到明细表 3-处理失败',
    retry_count     TINYINT      NOT NULL DEFAULT 0      COMMENT '重试次数',
    error_msg       VARCHAR(500) NULL                    COMMENT '处理失败原因',
    created_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '入队时间（Kafka Consumer 落库时间）',
    processed_at    DATETIME     NULL                    COMMENT '处理完成时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_publish_student (publish_id, student_id),
    INDEX idx_process_status (process_status),
    INDEX idx_publish_id     (publish_id),
    INDEX idx_created_at     (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='交卷流水暂存表（承载高并发交卷容灾；Kafka消费落库后再异步展开到 student_answer）';
```

**`answers_json` 格式说明：**

```json
{
  "submitAt": "2026-06-16T10:30:25.123Z",
  "answers": [
    { "questionId": 1001, "content": "A",           "type": 1 },
    { "questionId": 1002, "content": "A,C",         "type": 2 },
    { "questionId": 1003, "content": "这道题的解答...", "type": 5 }
  ],
  "clientIp": "192.168.1.100",
  "draftVersion": 3
}
```

**交卷处理流水线：**

```
前端打散交卷（学号 % 30 = 延迟秒数）
  │
  ├─① POST /exam/{id}/submit
  │    ├─ Redis 幂等键检查（exam:submit:{publishId}:{studentId}，TTL 30min）
  │    ├─ Redis 暂存答案（exam:answer:{publishId}:{studentId}，TTL 2h）
  │    └─ 发 Kafka: edu.exam.submit（返回200，毫秒级响应）
  │
  ├─② Kafka Consumer（并发度=10）平滑消费
  │    └─ INSERT INTO exam_submit_queue (process_status=0)
  │
  └─③ XXL-Job 异步处理（每分钟扫描 process_status=0 的记录）
       ├─ 解析 answers_json
       ├─ 批量 INSERT INTO student_answer
       └─ UPDATE exam_submit_queue SET process_status=2
```

## 15. 索引策略汇总

### 15.1 索引设计原则

```
原则 1  主键索引：BIGINT AUTO_INCREMENT，禁止 UUID 做主键
原则 2  唯一约束：使用 UNIQUE KEY，应用层捕获 DuplicateKeyException
原则 3  外键字段：必须建普通索引（InnoDB 不自动建外键列索引）
原则 4  ETL 对照字段（jwxt_*）：必须建独立普通索引，禁止全表线性扫描
原则 5  TEXT 字段：不建普通前缀索引，使用 FULLTEXT（WITH PARSER ngram）
原则 6  高频过滤字段：status / semester / created_at 等建普通索引
原则 7  低选择性字段不单独建索引（is_deleted / user_type 等仅 2~3 值）
原则 8  批量 INSERT 热点表（attendance）不建 UNIQUE KEY，幂等交给应用层
```

GRANT SELECT, INSERT, UPDATE ON edu_db.jwxt_id_mapping TO 'edu_jwxt'@'%';

-- 审计管理员（三员分立：只读审计库）
CREATE USER 'edu_auditor'@'%' IDENTIFIED BY '${EDU_AUDITOR_PWD}';
GRANT SELECT ON edu_audit_db.* TO 'edu_auditor'@'%';

-- 日志写入账号（只有 INSERT，无 UPDATE/DELETE，等保合规）
