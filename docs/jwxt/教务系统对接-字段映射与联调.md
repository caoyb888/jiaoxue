# 教务系统对接 — 字段映射与联调文档（S7-16）

> 配套实现：S7-08 三表逻辑 + ID 双向映射、S7-09 增量同步 ETL + 正方/强智适配层、S7-10 全量同步。
> 适用：正方（zhengfang）、强智（qiangzhi）教务系统。对接 `edu-jwxt`（端口 8093）。

---

## 1. 总体数据流

```
教务系统 API ──(JwxtDataSource 适配层)──> JwxtRecord(归一)
   │
   ├─> jwxt_raw_data        原始 JSON 暂存（append-only，处理后 90 天清理）
   ├─> jwxt_id_mapping      教务ID ↔ 本系统主键 双向唯一映射（增量判新增/更新）
   └─> 业务表(sys_user / sys_dept / course / class_room)   解析落库（联调期接入）
   全程记入 jwxt_sync_log（INCREMENTAL / FULL，成功/失败/计数/耗时）
```

适配层接口 `JwxtDataSource.fetchIncremental(dataType, since, offset, limit)` 分批拉取（防超时）；
`AbstractJwxtAdapter.toRecord(dataType, row)` 按厂商 `idField` 取主键并整行序列化为 `raw_json`。
当前生效厂商由 `jwxt.vendor`（`zhengfang`/`qiangzhi`/`stub`，dev 默认 `stub`）选择。

## 2. 数据类型与主键字段（适配层 idField）

`jwxt_raw_data.data_type` / `JwxtDataType` 取值与各厂商主键字段名：

| 数据类型 | 说明 | 正方主键字段 | 强智主键字段 | 落地业务表 |
|---------|------|------------|------------|-----------|
| `STUDENT` | 学生 | `xh`（学号） | `xsbh`（学生编号） | `sys_user`（user_type=1） |
| `DEPT` | 院系/专业 | `dwh`（单位号） | `bmbh`（部门编号） | `sys_dept` |
| `COURSE` | 课程 | `kch`（课程号） | `kcbh`（课程编号） | `course` |
| `CLASS` | 教学班 | `jxbid`（教学班ID） | `jxbbh`（教学班编号） | `class_room` |

> 主键字段定义见 `ZhengfangAdapter.ID_FIELDS` / `QiangzhiAdapter.ID_FIELDS`。
> 新增厂商：实现 `JwxtDataSource` 并提供各 `dataType` 的 `idField` 即可，编排层无需改动。
> 同步顺序 `JwxtDataType.SYNC_ORDER = [DEPT, COURSE, STUDENT, CLASS]`（院系/课程先行，便于学生/教学班的外键对照）。

## 3. 字段映射明细（教务 → 本系统业务表）

> 列名按「正方字段 / 强智字段」给出；本系统已为四类实体预留教务对照列
> （`jwxt_uid` / `jwxt_dept_id` / `jwxt_course_id` / `jwxt_class_id`）与自然键
> （`student_no` / `dept_code` / `course_code` / `class_code`）。

### 3.1 学生 STUDENT → `sys_user`（user_type=1）

| 本系统字段 | 正方字段 | 强智字段 | 说明 |
|-----------|---------|---------|------|
| `jwxt_uid` | `xh` | `xsbh` | 教务原始ID，ETL 对照键（亦写入 `jwxt_id_mapping`，data_type=USER） |
| `student_no` | `xh` | `xsbh` | 学号（唯一） |
| `real_name` | `xm` | `xm` | 姓名 |

| `dept_id` | `dwh` → 经映射 | `bmbh` → 经映射 | 先用院系教务ID 经 `jwxt_id_mapping(DEPT)` 解析为本系统 dept_id |
| `user_type` | 常量 1 | 常量 1 | 学生 |
| `status` | `zxzt`（在校状态） | `zt` | 映射为 0禁用/1正常 |

### 3.2 院系 DEPT → `sys_dept`

| 本系统字段 | 正方字段 | 强智字段 | 说明 |
|-----------|---------|---------|------|
| `jwxt_dept_id` | `dwh` | `bmbh` | 教务原始ID（映射 data_type=DEPT） |
| `dept_code` | `dwh` | `bmbh` | 院系编码（唯一对照码） |
| `dept_name` | `dwmc` | `bmmc` | 院系/专业名称 |
| `parent_id` | `sjdwh` → 经映射 | `sjbmbh` → 经映射 | 上级单位教务ID 经映射解析 |
| `dept_type` / `level` | `dwlb` | `bmlb` | 学院/系/专业/班级层级映射 |

### 3.3 课程 COURSE → `course`

| 本系统字段 | 正方字段 | 强智字段 | 说明 |
|-----------|---------|---------|------|
| `jwxt_course_id` | `kch` | `kcbh` | 教务原始ID（映射 data_type=COURSE） |
| `course_code` | `kch` | `kcbh` | 课程编码（唯一） |
| `course_name` | `kcmc` | `kcmc` | 课程名称 |
| `dept_id` | `kkdw` → 经映射 | `kkbm` → 经映射 | 开课院系教务ID 经映射解析 |
| `credit` | `xf` | `xf` | 学分 |
| `course_type` | `kclb` | `kclb` | 必修/选修/实验/实践映射 |
| `semester` | `xnxq` | `xnxq` | 学期，规整为 `2025-2026-1` |

### 3.4 教学班 CLASS → `class_room`

| 本系统字段 | 正方字段 | 强智字段 | 说明 |
|-----------|---------|---------|------|
| `jwxt_class_id` | `jxbid` | `jxbbh` | 教务原始ID（映射 data_type=CLASS） |
| `class_code` | `jxbmc`/`jxbid` | `jxbbh` | 教学班编码（唯一） |
| `class_name` | `jxbmc` | `jxbmc` | 教学班名称 |
| `course_id` | `kch` → 经映射 | `kcbh` → 经映射 | 课程教务ID 经映射(COURSE)解析 |
| `teacher_id` | `jsh` → 经映射 | `jsbh` → 经映射 | 教师教务ID 经映射(USER)解析 |
| `dept_id` | `kkdw` → 经映射 | `kkbm` → 经映射 | 开课院系经映射解析 |
| `semester` | `xnxq` | `xnxq` | 学期标识 |

## 4. ID 双向映射（`jwxt_id_mapping`）用法

双向唯一索引：`uk_type_jwxt_id (data_type, jwxt_id)` + `uk_type_local_id (data_type, local_id)`，O(1) 双向查找。

- `data_type` 取值：**USER / DEPT / COURSE / CLASS**（注意：学生在 raw_data 是 `STUDENT`，映射表用 **USER**；换算见 `JwxtSyncExecutor.mappingType`）。
- 增量同步判定：`JwxtMappingService.resolveLocalId(type, jwxtId)` 命中 → 更新；为空 → 新增。
- 反查：`resolveJwxtId(type, localId)`（成绩回传等需要教务学号时使用）。
- 写入：`saveMappings` / `upsert`（`ON DUPLICATE KEY UPDATE` 只刷 local_id/sync_log_id，幂等）。

```
教务记录 → toRecord → resolveLocalId(USER, xh)
    ├─ 命中：UPDATE sys_user … WHERE id = local_id
    └─ 未命中：INSERT sys_user → 取自增 id → upsert(USER, xh, id, syncLogId)
```

- [ ] 向学校教务处申请 **正方/强智测试沙盒账号 / API 凭据**（appKey、appSecret、接口 base-url、IP 白名单）。
- [ ] 确认接口协议与鉴权方式（正方常见 token/签名；强智常见 appKey+签名）。
- [ ] 索取四类数据（学生/院系/课程/教学班）的**接口字段说明书**，据此核对修订第 3 节映射。

### 5.2 配置（凭据经环境变量 / Nacos 注入，禁止硬编码入库）
```yaml
# edu-jwxt application.yml（占位，真实值经环境变量/Nacos）
jwxt:
  vendor: ${JWXT_VENDOR:stub}          # 联调切 zhengfang / qiangzhi
  sync:
    batch-size: ${JWXT_SYNC_BATCH:500} # 分批防超时
  zhengfang:
    base-url: ${JWXT_ZHENGFANG_URL:}   # 沙盒地址
  qiangzhi:
    base-url: ${JWXT_QIANGZHI_URL:}
```
> appKey/appSecret 等敏感凭据按 CLAUDE.md §9.2 统一存 Nacos（加密）并经环境变量注入，**禁止提交进 Git**。

### 5.3 适配层接入（待沙盒就绪）
- [ ] 在 `AbstractJwxtAdapter.fetchIncremental` 接入真实 HTTP 调用（当前 base-url 空时返回空、留有 TODO）。
- [ ] 按沙盒响应核对各 `dataType` 的 `idField` 与字段映射，修订第 3 节。
- [ ] 实现 raw_data → 业务表的解析落库（结合 `jwxt_id_mapping` 判新增/更新）。

### 5.4 联调验证步骤
1. `jwxt.vendor=zhengfang`（或 qiangzhi），配置沙盒 base-url。
2. 手动触发全量：`POST /api/v1/jwxt/sync/full`（管理员 token），完成后管理员收到微信/站内通知。
3. 核对 `jwxt_sync_log`（状态/计数/耗时）、`jwxt_raw_data`（原始 JSON）、`jwxt_id_mapping`（映射建立）。
4. 抽查业务表 `sys_user`/`sys_dept`/`course`/`class_room` 的 `jwxt_*` 对照列与字段正确性。
5. 配置 XXL-Job `jwxtIncrementalSync`（建议每日 02:00），验证增量只处理变更数据（`since` 取上次成功同步日期）。

### 5.5 合规要点（等保三级）
- 同步操作属写操作，全量接口已加 `@OperationLog`；同步过程全程记 `jwxt_sync_log`。
- 学生手机号等敏感字段落库须 AES-256 加密（`sys_user.phone_cipher`），日志脱敏（CLAUDE.md §9.2）。
- 教务凭据最小权限、网络白名单、不出网。
